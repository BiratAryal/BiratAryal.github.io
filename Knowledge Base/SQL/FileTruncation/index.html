    🗂 SQL Server Transaction Log (LDF) Truncation Guide

    This guide demonstrates how to safely truncate and shrink the transaction log file (LDF) of a SQL Server database using T-SQL. It's useful when the log file grows too large due to long-running transactions, replication issues, or a missing backup chain.

    ⚠️ Caution: Always back up your database before performing these actions, especially when changing recovery models.

    📌 1. Identify MDF and LDF File Names

    Run this query to get the logical names of the .mdf and .ldf files for your target database:

    USE <YourDatabaseName>;
GO

SELECT 
    name AS LogicalFileName,
    physical_name AS PhysicalFilePath,
    type_desc AS FileType
FROM 
    sys.database_files;

    Replace <YourDatabaseName> with the actual database name you're working on.

    🧾 2. Steps to Truncate and Shrink the LDF File

    Follow these steps after identifying the LDF file name.

    ✅ Step 1: Issue Checkpoints

    Flush dirty pages to disk before shrinking:

    USE <YourDatabaseName>;
GO

CHECKPOINT;
CHECKPOINT;

    🔁 Step 2: Switch Recovery Model to SIMPLE

    Temporarily switch to SIMPLE to allow automatic log truncation:

    ALTER DATABASE <YourDatabaseName>
SET RECOVERY SIMPLE;

    📉 Step 3: Shrink the Log File

    Shrink the transaction log file to a target size (e.g., 1,000 MB = 1 GB):

    DBCC SHRINKFILE (<YourLDFLogicalName>, 1000);  -- Size in MB
    • <YourLDFLogicalName> is the logical name of the log file (from Step 1).

    ♻️ Step 4: Revert to FULL Recovery Model

    Switch back to FULL recovery model:

    ALTER DATABASE <YourDatabaseName>
SET RECOVERY FULL;

    📝 Example (Replace with Your Values)

    USE MyDatabase;
CHECKPOINT;
CHECKPOINT;

ALTER DATABASE MyDatabase SET RECOVERY SIMPLE;
DBCC SHRINKFILE (MyDatabase_Log, 1000);
ALTER DATABASE MyDatabase SET RECOVERY FULL;

    🗂 SQL Server Data File (MDF) Truncation Guide

    This guide demonstrates how to safely truncate and shrink the transaction log file (LDF) of a SQL Server database using T-SQL. It's useful when the log file grows too large due to long-running transactions, replication issues, or a missing backup chain.

    ⚠️ Caution: Always back up your database before performing these actions, especially when changing recovery models.

    🧾 2. Steps to Truncate and Shrink the MDF File

CHECKPOINT;
CHECKPOINT;

    🔁 Step 2: Switch Recovery Model to SIMPLE

    ALTER DATABASE <YourDatabaseName>
SET RECOVERY SIMPLE;

    📉 Step 3: Shrink the MDF File

    Shrink the transaction log file to a target size (e.g., 1,000 MB = 1 GB):

CHECKPOINT;

DBCC SHRINKFILE (MyDatabase, 1000);
ALTER DATABASE MyDatabase SET RECOVERY FULL;

    🔐 Notes and Best Practices

    • Always take a full backup after switching back to FULL to reinitialize the log backup chain.
    • Avoid shrinking logs frequently—it can lead to fragmentation.
    • Consider analyzing why the log grew (e.g., long-running transactions, missing backups, replication backlog).

    🛠 Recommended Use Cases

    - Before configuring replication or migrating tables.
