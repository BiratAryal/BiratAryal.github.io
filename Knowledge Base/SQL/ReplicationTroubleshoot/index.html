    📘 MSSQL Replication Debugging Guide

    This guide documents the key SQL queries and commands used to monitor, troubleshoot, and optimize Transactional Replication in Microsoft SQL Server, particularly when CPU usage spikes due to replication activities.

    🧠 Overview

    This guide is helpful if: - You have configured transactional replication (evident by running Snapshot and Log Reader Agents). - You observe high CPU spikes, especially after adding new tables to replication. - You want to identify replication agent load, job schedules, query performance, and indexing issues.

    🔍 Check What Kind of Replication is Configured

    USE distribution;
GO

SELECT 
    pub.publisher_db,
    pub.publication,
    pub.publication_type,
    CASE pub.publication_type
        WHEN 0 THEN 'Transactional'
        WHEN 1 THEN 'Snapshot'
        WHEN 2 THEN 'Transactional with Updatable Subscriptions (deprecated)'
        WHEN 3 THEN 'Peer-to-Peer Transactional'
        WHEN 4 THEN 'Merge'
        ELSE 'Unknown'
    END AS replication_type
FROM dbo.MSpublications pub;
````

## ⏰ Check Replication Job Schedules

```sql
USE msdb;
GO

SELECT 
    j.name AS JobName,
    s.next_run_date,
    s.next_run_time
FROM dbo.sysjobs j
JOIN dbo.sysjobschedules js ON j.job_id = js.job_id
JOIN dbo.sysschedules s ON js.schedule_id = s.schedule_id
WHERE j.name LIKE '%snapshot%' OR j.name LIKE '%logreader%';

    Useful to identify if a Snapshot Agent or Log Reader Agent runs at specific times (e.g., 10:00 AM).

    🔎 Monitor High CPU Queries (Live)

    Run this at the time of high CPU (e.g., 10:00 AM):

    SELECT 
    r.session_id,
    r.status,
    r.command,
    r.cpu_time,
    r.total_elapsed_time,
    r.start_time,
    t.text AS query_text
FROM sys.dm_exec_requests r
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t
ORDER BY r.cpu_time DESC;

    🔄 Identify Active Replication Agent Queries

    SELECT 
    s.session_id,
    r.status,
    r.command,
    t.text AS query_text,
    r.cpu_time,
    r.start_time
FROM sys.dm_exec_requests r
JOIN sys.dm_exec_sessions s ON r.session_id = s.session_id
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t
WHERE s.program_name LIKE '%Repl%';

    This helps identify which replication agent jobs are actively consuming CPU or locking resources.

    📤 Monitor Pending Replication Commands (Latency)

    EXEC sp_replmonitorsubscriptionpendingcmds 
    @publisher = 'YourPublisherName', 
    @publisher_db = 'YourPublisherDB',
    @publication = 'YourPublicationName',
    @subscriber = 'YourSubscriberName',
    @subscriber_db = 'YourSubscriberDB';

    Shows how many commands are waiting to be replicated. A large number may indicate a bottleneck.

    🧱 Analyze the Replicated Table

    -- Table size
EXEC sp_spaceused 'YourTableName';

-- Row count
SELECT COUNT(*) FROM YourTableName;

-- Indexes on the table
EXEC sp_helpindex 'YourTableName';

    Ensure the table has a primary key and sufficient indexing to avoid heavy log scanning.

    ⚠️ View Replication Agent Errors & History

    USE distribution;
GO

SELECT 
    h.agent_id,
    a.name AS agent_name,
    h.run_status,
    h.run_duration,
    h.run_date,
    h.run_time,
    h.[message]
FROM dbo.MSdistribution_history h
JOIN dbo.MSdistribution_agents a ON h.agent_id = a.id
ORDER BY h.run_date DESC, h.run_time DESC;

    run_status:

    • 1 = Success
    • 2 = Retry
    • 3 = Failure
    • 4 = Canceled
    • 5 = In Progress

    📋 View Agent Jobs Manually

    -- List all replication-related SQL Agent jobs
SELECT name FROM msdb.dbo.sysjobs WHERE name LIKE '%Repl%';

    -- Job history for specific job
EXEC msdb.dbo.sp_help_jobhistory 
    @job_name = 'YourJobNameHere',
    @mode = 'FULL';

    📂 Read SQL Server Agent Logs

    -- Read the latest SQL error logs (including replication-related logs)
EXEC xp_readerrorlog;

    You can also check these logs via SSMS under SQL Server Agent → Error Logs.

    🛠 Recommendations for Optimization

    • Avoid daily snapshots unless required.
    • Schedule heavy replication tasks during off-peak hours.
    • Use proper indexes on frequently updated columns in replicated tables.
    • If one table causes spikes, consider moving it to a separate publication.

    📅 Optional: Log High-CPU Queries Automatically

    You can schedule this query to run at 10:00 AM daily to track high CPU queries:

    SELECT 
    r.session_id,
    r.cpu_time,
    r.total_elapsed_time,
    t.text AS query_text,
    r.start_time
INTO dbo.HighCpuQueries_Log
FROM sys.dm_exec_requests r
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t
WHERE r.cpu_time > 1000;

    🧼 Clean and Maintain Replication

    • Remove unnecessary subscriptions or publications.
    • Reinitialize only if required.
    • Periodically clean the distribution database if it grows too large.
