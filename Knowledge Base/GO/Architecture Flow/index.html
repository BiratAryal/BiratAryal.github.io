    🧭 Big Picture: How Bubble Tea Works

    Bubble Tea uses a Model-Update-View (MUV) architecture — a variant of The Elm Architecture. This helps you write clean, reactive terminal apps.

    🔁 The Event Loop in Bubble Tea

    Text Only
    User Input / Event
       ↓
      Msg
       ↓
  Update(model, msg)
       ↓
 New Model + Cmd
       ↓
   View(model)
       ↓
Render (Terminal Output)

    Now let’s go through each of these in depth:

    🔹 1. User Input / Events

    ❓ What is this?
    This includes:

    • Keyboard input (like pressing ↑ or ↓)

    • Mouse events

    • Timers (like a clock tick every second)

    • Network responses (if you're fetching data)

    🎯 Goal: Detect that something happened.

    🧠 You don’t process the input directly — instead, Bubble Tea transforms it into a Msg.

    🔹 2. Msg (Message)

    ❓ What is this?
    It’s a type or struct that represents what happened.

    You define a custom Msg type to represent user actions or system events.

    ✅ Example:

    Go
    type Msg string

const (
    MsgUp   Msg = "up"
    MsgDown Msg = "down"
)

    Bubble Tea sends this Msg into the next stage.

    🔹 3. Update(model, msg)

    ❓ What is this?
    The Update function handles logic.

    You write logic inside Update to:

    • Change the state (i.e., update the model)

    • Optionally trigger side effects like timers or network calls (via Cmd)

    ✅ Example:

    Go
    func (m model) Update(msg tea.Msg) (model, tea.Cmd) {
    switch msg := msg.(type) {
    case tea.KeyMsg:
        switch msg.String() {
        case "up":
            m.cursor--
        case "down":
            m.cursor++
        }
    }
    return m, nil
}

    🎯 Goal: Return a new model (updated state) and optionally a command (Cmd).

    🔹 4. New Model + Cmd

    ✅ Model: The new application state after handling the message.

    ✅ Cmd (Command): Tells Bubble Tea to run something asynchronously and return a new Msg later.

    Common use cases for Cmd:

    • Wait for a timer and send a tick

    • Make a network request

    • Read a file

    • Animate something

    ✅ Example:

    Go
    return m, tea.Tick(time.Second, func(t time.Time) tea.Msg {
    return tickMsg(t)
})

    🔹 5. View(model)

    ❓ What is this?
    It’s a function that takes your model (state) and returns a string to render in the terminal.

    🎯 Goal: Turn state into user interface.

    ✅ Example:

    Go
    func (m model) View() string {
    s := "Choose an option:\n\n"
    for i, choice := range m.choices {
        cursor := " " // no cursor
        if m.cursor == i {
            cursor = ">" // highlight
        }
        s += fmt.Sprintf("%s %s\n", cursor, choice)
    }
    return s
}

    🔄 Loop Back

    After rendering, Bubble Tea waits for the next event or message, and the cycle repeats.

    🎨 Visual Flow Summary

    Text Only
    1. User presses key          ⌨️
2. Bubble Tea converts to →  Msg
3. Msg sent to →             Update()
4. Update changes →          Model + optional Cmd
5. View() renders →          New UI based on state
6. Back to waiting for event...

    🔍 Real Example Flow

    Let’s say the user presses the down arrow key.

    Step-by-step:

    1. Key press: "down"

    2. Converted to: tea.KeyMsg{Type: KeyDown}

    3. Update() sees the "down" key and increments m.cursor

    4. New model with updated cursor

    5. View() highlights the next item in the list

    6. Terminal redraws the new screen

    🧠 Why This Architecture is Powerful:

    • Separation of Concerns: View is only for display. Update is only for logic.

    • Testable: You can test each function easily.

    • Async Friendly: Cmd makes async logic simple without goroutines everywhere.

    • Declarative UI: You describe what the UI should look like based on state.
