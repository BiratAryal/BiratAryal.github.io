    🧠 Git Merging Strategies - Detailed Documentation

    Comprehensive guide to Git merge strategies, including definitions, command examples, editor behavior, commit history implications, and practical workflows.

    🧭 Introduction

    Git merge strategies determine how changes from one branch are integrated into another. The right strategy depends on whether you want to retain detailed history, squash it, or ignore it entirely.

    ❓ Why Merging Strategies Matter

    • 🔍 Traceable project history

    • 🧘‍♀️ Clean, readable logs

    • 🛠️ Efficient conflict resolution

    • 🤝 Better team collaboration

    🔀 Common Git Merging Strategies

    1. Recursive (Default)

    Usage: Most common; works with two branches that have diverged.

    Bash
    git merge feature/login

    2. Resolve

    Usage: Simple, two-parent merge; mostly deprecated in favor of recursive.

    Bash
    git merge -s resolve feature/login

    3. Octopus

    Usage: Merge multiple branches (no conflicts allowed).

    Bash
    git merge -s octopus feature1 feature2 feature3

    4. Ours

    Usage: Use current branch's changes and discard the other.

    Bash
    git merge -s ours experiment-branch

    5. Subtree

    Usage: Merging unrelated projects while keeping their directory structures.

    Bash
    git merge -s subtree project-b

    6. Squash ✅

    Usage: Flatten all commits from a feature branch into one commit.
    Great for clean history and PR-based workflows.

    Bash
    git merge --squash feature/checkout-page

    🛠 What happens:

    • Git collects all commits from the feature branch.

    • The code from all those commits is merged into your working directory.

    • A single commit is created (but not yet committed).

    You’ll see a text editor (usually Vim or VSCode) open with something like:

    Text Only
    # This is a combination of 5 commits.
# The first commit's message is:
Add checkout page UI

# The 2nd commit message is:
Implement discount logic

# The 3rd commit message is:
Refactor checkout controller

# Please enter the commit message for your changes.

    📝 You can edit this message, summarizing all changes in a single commit:

    Text Only
    Add complete checkout functionality

- Added UI for checkout
- Implemented coupon/discount logic
- Refactored controller logic

    Then commit it manually:

    Bash
    git commit

    🔁 Final result: One new commit is added to the main branch — a flattened, clean version of the entire feature.

    🧩 Pro Tip:

    • Combine with git rebase -i to refine history before squashing.

    • Use in PR-based workflows for atomic merges.

    ⏩ Fast-forward vs. No Fast-forward

    Fast-forward (--ff)

    • Pointer just moves forward (no merge commit).
    Strategy Description
    --ff Moves the branch pointer ahead—no merge commit
    --no-ff Forces a merge commit—keeps merge info
    Bash
    git merge --ff feature

    No Fast-forward (--no-ff)

    • Always creates a merge commit.
    Bash
    git merge --no-ff feature

    🌍 Real-world Use Cases

    Scenario Strategy Why
    Feature branch merge --no-ff Keeps history of the feature
    Hotfix multiple branches octopus Merge all hotfixes at once
    Discard experimental code ours Keep current state, mark merge complete
    Vendor code or repo fusion subtree Preserve separate history in a sub-dir
    Simplify history for PR --squash Single meaningful commit on main branch
    Rewriting messy local commits rebase -i Makes feature history clean

    📊 Merge vs Squash vs Rebase – Comparison

    Feature / Goal merge --squash rebase octopus ours
    Preserve commit history ✅ Yes ❌ No (one commit) ✅ Yes (rewritten) ✅ Yes ❌ No
    Creates merge commit ✅ Yes ❌ No ❌ No ✅ Yes ✅ Yes (but fake)
    Used in PR workflows ✅ Common ✅ Very common ⚠️ Sometimes ⚠️ Rarely ❌ No
    Conflict resolution ✅ Supported ✅ At squash time ✅ During rebase ❌ Not supported ❌ Discards changes
    Interactive history cleanup ❌ No ⚠️ Limited ✅ Yes (with -i) ❌ No ❌ No
    Automated multi-branch ❌ No ❌ No ❌ No ✅ Yes ❌ No
    Final commit behavior Merge + all commits One single commit Rewritten commits Merge of many “Keep ours” only
    Best for Team collaboration Clean history Linear development Release branches Ignoring other branch

    ⚙️ Workflow Recommendations

    GitHub / GitLab Merge Requests (PR Model)

    Bash
    # Checkout main
git checkout main

# Merge feature cleanly
git merge --squash feature/login
git commit -m "Add user login functionality"

    Use --squash in conjunction with CI pipelines and PR reviews to reduce commit noise and keep history readable.

    🚧 Interactive Rebase

    Bash
    git rebase -i HEAD~5
# Reorder, squash, or fixup commits before merging

    ✅ Best Practices

    • 📌 Use --no-ff when you want to preserve merge context.

    • 🧹 Use --squash to clean messy commits before merging to main.

    • 🔀 Avoid octopus if there's any chance of a conflict.

    • 🚫 Use ours sparingly—you're discarding someone’s work.

    • 🧰 Document your team strategy in CONTRIBUTING.md.
