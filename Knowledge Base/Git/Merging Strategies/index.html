    üß≠ Introduction

    Git merge strategies determine how changes from one branch are integrated into another. The right strategy depends on whether you want to retain detailed history, squash it, or ignore it entirely.

    ‚ùì Why Merging Strategies Matter

    • üîç Traceable project history

    üîÄ Common Git Merging Strategies

    1. Recursive (Default)

    Usage: Most common; works with two branches that have diverged.

    Bash
    git merge feature/login

    2. Resolve

    Usage: Simple, two-parent merge; mostly deprecated in favor of recursive.

    Bash
    git merge -s resolve feature/login

    3. Octopus

    Usage: Merge multiple branches (no conflicts allowed).

    Bash
    git merge -s octopus feature1 feature2 feature3

    4. Ours

    Usage: Use current branch's changes and discard the other.

    Bash
    git merge -s ours experiment-branch

    5. Subtree

    Usage: Merging unrelated projects while keeping their directory structures.

    Bash
    git merge -s subtree project-b

    6. Squash ‚úÖ

    Usage: Flatten all commits from a feature branch into one commit.
    Great for clean history and PR-based workflows.

    Bash
    git merge --squash feature/checkout-page

    üõ† What happens:

    • Git collects all commits from the feature branch.

    • The code from all those commits is merged into your working directory.

    • A single commit is created (but not yet committed).

    You‚Äôll see a text editor (usually Vim or VSCode) open with something like:

    Text Only
    # This is a combination of 5 commits.
# The first commit's message is:
Add checkout page UI

# The 2nd commit message is:
Implement discount logic

# The 3rd commit message is:

# Please enter the commit message for your changes.

    üìù You can edit this message, summarizing all changes in a single commit:

    Text Only

- Added UI for checkout
- Implemented coupon/discount logic
- Refactored controller logic

    Then commit it manually:

    Bash
    git commit

    üîÅ Final result: One new commit is added to the main branch ‚Äî a flattened, clean version of the entire feature.

    üß© Pro Tip:

    • Combine with git rebase -i to refine history before squashing.

    • Use in PR-based workflows for atomic merges.

    ‚è© Fast-forward vs. No Fast-forward

    Fast-forward (--ff)

    • Pointer just moves forward (no merge commit).
    Strategy Description
    --ff Moves the branch pointer ahead‚Äîno merge commit
    --no-ff Forces a merge commit‚Äîkeeps merge info
    Bash
    git merge --ff feature

    No Fast-forward (--no-ff)

    • Always creates a merge commit.
    Bash
    git merge --no-ff feature

    üåç Real-world Use Cases

    Scenario Strategy Why
    Feature branch merge --no-ff Keeps history of the feature
    Hotfix multiple branches octopus Merge all hotfixes at once
    Discard experimental code ours Keep current state, mark merge complete
    Vendor code or repo fusion subtree Preserve separate history in a sub-dir
    Simplify history for PR --squash Single meaningful commit on main branch
    Rewriting messy local commits rebase -i Makes feature history clean
